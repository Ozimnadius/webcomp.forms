# webcomp.forms — формы на базе инфоблоков

Кастомный модуль 1С-Битрикс для управления формами через инфоблоки. Каждая форма — обычный инфоблок типа `forms`: свойства инфоблока описывают поля публичной формы, а элементы того же инфоблока хранят отправленные заявки. Модуль сам готовит почтовую инфраструктуру (тип события + шаблон письма) и предоставляет компонент для вывода формы на сайте.

> ⚠️ **Важно:** Перед установкой/удалением модуля сделать резервную копию (модуль пока в альфа-тесте).

## Возможности

- Тип инфоблоков `forms` создаётся при установке модуля.
- Поля формы описываются свойствами кастомного типа `formbuilder` (тип контрола, подпись, placeholder, обязательность, варианты значений).
- При создании/изменении инфоблока формы автоматически создаётся и синхронизируется тип почтового события `WEBCOMP_FORM_<CODE>` и почтовый шаблон.
- Компонент `webcomp:form` выводит форму, валидирует отправку, сохраняет заявку элементом инфоблока и ставит письмо в очередь.
- Поддерживаются обычная отправка (с Post/Redirect/Get) и AJAX-отправка (JSON-ответ, без штатного AJAX-режима Битрикса).
- В админке на странице свойств инфоблока `forms` выбор типа свойства ограничен «Полем формы».
- Деинсталляция удаляет все данные модуля: инфоблоки форм, тип `forms`, почтовые события и шаблоны, опции, скопированные файлы.

## Требования

- 1С-Битрикс с модулями `main` и `iblock`.
- PHP 8.0+.

## Установка

Стандартно: «Marketplace → Установленные решения» → модуль `webcomp.forms` → «Установить». При установке модуль:

1. Копирует ассеты в `/bitrix/css/webcomp.forms/` и `/bitrix/js/webcomp.forms/`, компонент — в `/local/components/webcomp/form/` (или `/bitrix/components/`, если модуль лежит в `/bitrix/modules/`).
2. Регистрирует обработчики событий `iblock` и `main`.
3. Создаёт тип инфоблоков `forms` (ошибка создания прерывает установку с текстом причины).

## Как создать форму

1. Создайте инфоблок в типе «Формы». Заполните символьный код (латиницей) — он станет частью имени почтового события; при пустом или нелатинском коде используется ID инфоблока.
2. Добавьте свойства — каждое свойство является полем формы. Тип свойства — «Поле формы». В настройках свойства задайте:
   - **Тип поля формы**: `text`, `email`, `tel`, `url`, `number`, `date`, `textarea`, `select`, `checkbox`, `radio`;
   - **Подпись поля** и **Placeholder**;
   - **Обязательное** — серверная и HTML5-валидация;
   - **Варианты значений** (для `select` / `checkbox` / `radio`) — пары «значение / текст».
3. Разместите компонент на странице:

```php
<?php $APPLICATION->IncludeComponent(
    'webcomp:form',
    '',
    [
        'IBLOCK_ID' => 7,            // или 'IBLOCK_CODE' => 'feedback'
        'USE_AJAX' => 'Y',
        'SUBMIT_TEXT' => 'Отправить',
    ]
); ?>
```

Заявки сохраняются элементами того же инфоблока; ссылка на заявку приходит в письме (макрос `#ADMIN_RESULT_URL#`).

## Параметры компонента `webcomp:form`

| Параметр | Описание | По умолчанию |
|---|---|---|
| `IBLOCK_ID` | ID инфоблока формы | — |
| `IBLOCK_CODE` | Символьный код формы, если ID не задан | — |
| `USE_AJAX` | AJAX-отправка через fetch (`Y`/`N`) | `Y` |
| `SUBMIT_TEXT` | Текст кнопки отправки | `Отправить` |
| `DISPLAY_MODE` | Режим отображения: `inline` / `popup` | `inline` |
| `BUTTON_TEXT` | Текст кнопки-триггера попапа | `Открыть форму` |
| `BUTTON_CLASS` | CSS-классы кнопки-триггера попапа | `btn btn-primary` |

Формат AJAX-ответа:

```json
{ "success": true, "resultId": 123, "errors": {}, "values": {} }
```

При ошибках `errors` содержит пары «код поля → текст ошибки» (ключ `SYSTEM` — общая ошибка).

## Попап-режим (`DISPLAY_MODE = popup`)

```php
<?php $APPLICATION->IncludeComponent(
    'webcomp:form',
    '',
    [
        'IBLOCK_ID' => 7,
        'DISPLAY_MODE' => 'popup',
        'BUTTON_TEXT' => 'Заказать звонок',
    ]
); ?>
```

Как это работает:

- на странице рендерятся только кнопка и пустой нативный `<dialog>` — **без запросов к инфоблоку и без sessid в разметке**, поэтому страница с кнопкой безопасно кэшируется любым кэшем (включая композит);
- при первом открытии попапа HTML формы лениво загружается отдельным запросом (`webcomp_form_render=Y` + `PARAMS_HASH`) и приносит **свежий sessid текущей сессии**; фрагмент кэшируется в памяти страницы — повторное открытие мгновенно;
- отправка из попапа — обычный AJAX-путь компонента (`USE_AJAX` в попап-режиме принудительно `Y`);
- модалка — нативный `<dialog>`: закрытие по крестику, клику на подложку и ESC, без зависимости от Bootstrap JS;
- открыть попап может любая кнопка темы: достаточно атрибута `data-webcomp-form-open="<id диалога>"` (id виден в разметке кнопки компонента);
- fallback без JS: кнопка неактивна; если JS-отправка не удалась и форма ушла обычным POST — компонент деградирует до инлайн-страницы с формой, а после успеха возвращает на страницу с баннером у кнопки.

## Почтовые события

Для каждой формы создаётся тип события `WEBCOMP_FORM_<CODE>` и один почтовый шаблон для сайтов инфоблока. Доступные макросы: служебные `#FORM_NAME#`, `#FORM_CODE#`, `#FORM_ID#`, `#RESULT_ID#`, `#DATE_CREATE#`, `#SITE_NAME#`, `#SERVER_NAME#`, `#ADMIN_RESULT_URL#` плюс по макросу `#<КОД_СВОЙСТВА>#` на каждое поле (полный список — в описании типа события в админке).

Правила синхронизации:

- описание типа события обновляется при каждом изменении формы;
- **тело шаблона генерируется один раз при создании** и дальше не перезаписывается — ручные правки письма сохраняются; новые поля админ добавляет в шаблон сам (макросы видны в описании события);
- привязка шаблона к сайтам синхронизируется с привязкой инфоблока;
- связь «форма → событие → шаблон» хранится в опциях модуля (`mail_binding_<ID>`), поэтому переименование кода инфоблока переносит событие и шаблон без создания дублей;
- значения полей экранируются перед подстановкой в HTML-письмо;
- сбой отправки письма не считается ошибкой отправки формы: заявка уже сохранена, ошибка пишется в журнал (`AddMessage2Log`, лог-метка `webcomp.forms`).

## Защита отправки

- `bitrix_sessid_post()` + серверная проверка `check_bitrix_sessid()`;
- `PARAMS_HASH` — сверка параметров компонента при отправке;
- серверная валидация: обязательность, принадлежность значений списку вариантов, лимит длины (5000 символов), нормализация типов значений из POST;
- Post/Redirect/Get для не-AJAX отправки (повтор по F5 не создаёт дубль).

Captcha, honeypot и rate limit в MVP не входят (см. бэклог в `docs/forms-module.md`).

## Архитектура

| Класс | Ответственность |
|---|---|
| `Form\IblockTypeInstaller` | Создание/удаление типа инфоблоков `forms` и инфоблоков форм |
| `Form\FormRepository` | Чтение формы, её полей и привязок к сайтам (канонический читатель) |
| `Form\MailEventSynchronizer` | Создание/синхронизация/удаление типа события и шаблона, привязки в опциях |
| `Form\ResultService` | Валидация отправки, сохранение заявки, постановка письма в очередь |
| `EventHandlers\IblockHandler` | События инфоблоков: синхронизация почты при изменениях, зачистка при удалении |
| `EventHandlers\AdminHandler` | Подключение админского JS, ограничивающего типы свойств для `forms` |
| `PropertyTypes\FormBuilder` | Кастомный тип свойства `formbuilder` (настройки поля, админский рендер) |

Компонент: `install/components/webcomp/form` (класс `WebcompFormComponent`, view-model готовится в `result_modifier.php`, шаблон — чистое представление, `script.js` подключается автоподключением шаблона).

## Осознанные решения

- Строки интерфейса захардкожены по-русски — проект русскоязычный, локализация через `Loc` не делалась (кроме установщика).
- Bootstrap 5 подключается с CDN jsdelivr в шаблоне компонента; при работе без CDN/со строгим CSP положите локальную копию в `install/css` и поменяйте подключение в шаблоне.
